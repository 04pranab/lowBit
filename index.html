01 — Overview

What LowBIt is

LowBIt is not a single project, and this site is not its product. It's a personal benchmarking initiative — an aggregation of independently maintained benchmark packages that share one specification, one methodology, and one registry.

Each benchmark is intentionally small: a single kernel, a fixed set of workloads, a golden reference output. Nothing here is a framework you install — every package is a standalone repository you can clone, build, and validate on its own.

This repository is the index. It hosts no benchmark source code, only the rules a benchmark must follow to join the suite, and the growing list of benchmarks that do.

02 — Philosophy

Why it's built this way

  • ▸
    Build small, composable pieces.

    Every benchmark does one thing — one kernel, one measurement — well.

  • ▸
    Prefer clarity over completeness.

    A readable, honest harness beats a clever one nobody can audit.

  • ▸
    Learn by implementation.

    Theory is checked against a build that either runs deterministically or doesn't.

  • ▸
    Allow unfinished and experimental work.

    Some entries in the registry are planned, not shipped. That's shown, not hidden.

  • ▸
    Optimize for understanding, not speed.

    A number without a reproducible method behind it isn't a result.

04 — Methodology

What "validated" means here

The discipline follows SPEC CPU®'s determinism/portability/validation tradition, with an Icicle-inspired top-down approach to microarchitectural analysis where hardware performance counters are actually available. Every benchmark self-reports against the same seven principles:

Determinismverified

Fixed-seed RNG, fixed reduction order, no unordered containers. Repeat runs are bit-identical.

Portabilityverified

ISO-standard language only. No required intrinsics or inline assembly. Builds unmodified across target architectures.

Diversitypartial

Coverage grows kernel by kernel. Gaps are flagged in each package's metadata.yaml, never left silent.

Validationverified

Golden-checksum references per architecture, regenerable, checked by a self-contained --validate mode.

Longevityverified

Minimal, stable dependencies. A benchmark should still build in five years without an archaeology project.

Analysispartial

Top-down microarchitectural analysis where a PMU exists. Declines to fabricate numbers where it doesn't.

User ownershipverified

Explicit per-package licensing. Nothing generated here is public by default.
