# LowBIt Benchmark Specification

Version 1.0 — governs every package listed in [BENCHMARKS.md](BENCHMARKS.md).

This document defines what a repository must contain to be admitted to the
LowBIt registry. It is deliberately narrow: it constrains *shape and
discipline*, not implementation language, kernel choice, or target
hardware. A package that satisfies this spec is comparable to every other
package in the suite without anyone having to read its source first.

`ml-transformer-attn_r` (`LB-ML-001`) is the reference implementation of
this spec. Where this document is ambiguous, that package's actual layout
is the tiebreaker.

---

## 1. Required layout

```
<package>/
├── README.md            benchmark card + quickstart + results snapshot
├── LICENSE               ownership terms for this package specifically
├── benchmark.json          machine-readable descriptor (§3)
├── metadata.yaml             provenance + compliance checklist (§4)
├── input/                      declarative workload recipes
├── reference/                    golden reference outputs, per architecture
├── scripts/                        build / run / validate / regen entry points
└── docs/                             design rationale
```

A package missing any of these is **not eligible** for the registry,
regardless of how correct or interesting its benchmark is.

## 2. The benchmark card (README.md)

Every `README.md` opens with a fenced block in this exact field set, in
this order:

```
Benchmark ID:      LB-<DOMAIN>-<NNN>
Kernel:            <human name>
Domain:            <Machine Learning | Memory | Compute | Cryptography | Graph | I/O>
Category:          <e.g. Dense Linear Algebra>
Operations:        <comma-separated>
Language:          <e.g. C++17>
Deterministic:     Yes | No
Reference Output:  Yes | No
Validated:         Yes | No
Architectures:     <comma-separated ISA targets>
Difficulty:        Low | Medium | High
```

This is what the registry card on the site renders directly — if a field
here is wrong, it's wrong everywhere downstream.

## 3. `benchmark.json` — required fields

```jsonc
{
  "suite": "LowBIt",
  "id": "LB-<DOMAIN>-<NNN>",          // must match LOWBIT_REGISTRY.md exactly
  "name": "package-directory-name",
  "version": "semver",
  "description": "one sentence, no marketing",
  "kernel": {
    "type": "string",
    "phase": "inference | training | n/a",
    "operations": ["..."]
  },
  "workloads": [
    { "name": "test", "recipe": "input/test.json", "purpose": "CI / smoke test" }
  ],
  "build": {
    "system": "e.g. CMake >= 3.16",
    "standard": "language standard, ISO-only preferred",
    "targets": { "<name>": "<build command>" }
  },
  "validation": {
    "determinism": "what bit-identical guarantee is claimed, and its scope",
    "accuracy_gate": "if the benchmark has a lower-precision path"
  },
  "reference_outputs": "path or glob to golden files",
  "metrics_reported": ["mean_ms", "..."],
  "license": "see LICENSE",
  "owner": "string"
}
```

A `test` workload — small enough to run in CI in well under a minute — is
mandatory. `ref` and any larger stress workloads are recommended but not
required.

## 4. `metadata.yaml` — compliance checklist

`metadata.yaml` must self-report against all seven methodology principles
in [methodology/](methodology/) (determinism, portability, diversity /
representativeness, validation, longevity, analysis, user ownership), each
as one of:

```yaml
<principle>:
  status: verified | partial | not-met
  notes: >
    One paragraph, plain language, stating what was actually checked and
    what wasn't. A "partial" or "not-met" status with an honest note is
    preferred over a "verified" that overstates what was tested.
```

`known_gaps` is a required top-level list, even if empty. Silence on a
limitation is treated as a spec violation; a documented gap is not.

## 5. Workloads are data, not code

Workload shapes belong in `input/*.json`, loaded at runtime. A new
board-tuned size is a new JSON file — never a source change and never a
recompile. Each recipe is a **generation instruction** (seed + parameters),
not a data dump, so it stays small and version-controllable while still
being the complete, reproducible specification of what the benchmark ran
on.

## 6. Reference outputs

`reference/` holds golden, checksummed outputs, generated by actually
running the benchmark — never hand-written. One golden file per
architecture per workload, **not** a single universal file, because bit-
identical guarantees are not assumed to hold across architectures unless a
package explicitly demonstrates and documents that they do (see
`docs/DESIGN.md` in `ml-transformer-attn_r` for a worked example of where
that guarantee holds — integer paths — and where it doesn't — unconstrained
floating point).

`reference/README.md` documents, per file: what generated it, what it's
checked against, and any cross-architecture caveats.

## 7. Scripts

At minimum: a native build, a `--validate` (or equivalent) self-check mode
that re-runs and requires bit-identical output within one build, and a
reference-regeneration script. Cross-compilation and emulation scripts are
encouraged wherever the target architecture isn't the dev host.

## 8. Docs

`docs/DESIGN.md` explains *why*, not just *what*: why this kernel and not
an adjacent one, what was actually verified in the most recent working
session (with real numbers or checksums, not placeholders), and — just as
important — what was **not** verified and why, stated plainly rather than
omitted. Fabricated numbers for unavailable measurements (e.g. timing under
a non-cycle-accurate emulator) are treated as a spec violation.

## 9. Admission to the registry

1. Reserve the next free ID in your domain in `LOWBIT_REGISTRY.md` /
   [BENCHMARKS.md](BENCHMARKS.md).
2. Satisfy §1–§8 in the package's own repository.
3. Add one entry to the registry with a link once the package is public.

A reserved ID with no package yet is listed as **planned**, not omitted —
see the [Roadmap](ROADMAP.md).
